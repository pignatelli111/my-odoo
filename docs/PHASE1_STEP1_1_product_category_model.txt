Phase 1 ‚Äî Step 1.1: Product & category model (runbook)
======================================================

Goal: Map Suburban **business families** (materials, accessories, gaskets, coils,
kits, services, subcontracting, logistics) to **Odoo product categories** and
**attributes** so they drive behaviour (defaults, routes, reporting), not only
labels on a form.

Prerequisite: Core apps on (0.3); accounting skeleton (0.2); migration plan (0.4)
understood.

A) Vocabulary: ‚Äúfamily‚Äù vs Odoo objects
---------------------------------------
| Client concept (family)   | Primary Odoo object        | Also use |
|---------------------------|----------------------------|----------|
| Standard materials        | `product.category` + product | Variants if needed |
| Specialist components     | category + product         | Attributes for ‚Äúsystem‚Äù |
| Accessories               | category + product         | Often purchase = ordered |
| Gaskets                   | category + product         | UoM m / ml / pcs |
| Coils / continuous        | category + product         | UoM length; loss % later |
| Packs / kits              | category + **Kit** product or BOM | Decide kit vs phantom BOM |
| External services         | **Service** type product   | No stock |
| Technical consulting      | Service product            | Timesheet optional later |
| Subcontracted work        | Service or **Consu** + vendor | Route subcontract if used |
| Logistics / transport     | Service product            | Link to project cost later |

D) Defaults per category (behaviour, not only reporting)
-------------------------------------------------------
For each **leaf category**, agree and set where Odoo allows:

[ ] **Costing method** (standard / FIFO) ‚Äî often one policy per company first.
[ ] **Routes** (Buy, MTO, Manufacture) ‚Äî at least Buy vs MTO for stock items.
[ ] **Income / expense accounts** (on category or on product category properties)
    so SAL / invoices propose correct accounts.
[ ] **Purchase / sales UoM** defaults where the family always uses the same unit.

E) Link to SBU custom flows (forward compatibility)
---------------------------------------------------
[ ] Categories used on **BOM lines** (`sbu.estimate.bom.line`) should reference
    `product.product` rows that sit in the right category ‚Äî no orphan ‚Äúmisc‚Äù.
[ ] **Purchase requests** (RDA-style): later RFQ grouping will use **product /
    category** filters ‚Äî keep category tree aligned with buyer mental model.
